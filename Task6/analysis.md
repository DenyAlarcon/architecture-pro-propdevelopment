# Отчёт по результатам анализа Kubernetes Audit Log

## Подозрительные события

1. Доступ к секретам:
- Кто: `system:serviceaccount:secure-ops:monitoring`.
- Где: `kube-system`, ресурс `secrets`.
- Почему подозрительно: сервисный аккаунт из прикладного namespace пытается читать системные секреты.

2. Привилегированные поды:
- Кто: пользователь из поля `user.username` события создания `privileged-pod`.
- Комментарий: `privileged: true` снимает часть контейнерной изоляции и повышает риск компрометации узла.

3. Использование kubectl exec в чужом поде:
- Кто: пользователь из поля `user.username` события `subresource=exec`.
- Что делал: выполнял `kubectl exec` в поде из `kube-system` (например, CoreDNS).

4. Создание RoleBinding с правами cluster-admin:
- Кто: пользователь из поля `user.username` события создания `RoleBinding/escalate-binding`.
- К чему привело: сервисному аккаунту `monitoring` выданы права `cluster-admin` (эскалация привилегий).

5. Удаление audit-policy.yaml:
- Кто: `admin` (имперсонация через `--as=admin`).
- Возможные последствия: ослабление/потеря аудита и снижение обнаружения последующих атак.

## Вывод

Сценарий содержит признаки компрометации: доступ к секретам, эскалация привилегий, запуск привилегированного пода и попытка воздействовать на аудит. Нужно ужесточить RBAC, запретить `privileged` (PodSecurity/Gatekeeper), ограничить impersonation и настроить алерты на события `secrets`, `exec`, `cluster-admin rolebinding`, `audit-policy`.
